# 中文 AI 技术文章风格规范

这是 `write-ai-article` skill 写每一篇文章前必读的硬约束。风格综合自 [@dotey 宝玉](https://x.com/dotey)、[Simon Willison](https://simonwillison.net/)、[Andrej Karpathy](https://karpathy.ai/)、[Lilian Weng](https://lilianweng.github.io/)、[Jay Alammar](https://jalammar.github.io/)、[Chip Huyen](https://huyenchip.com/)。

## 十条硬规则

### 1. 开头直接抛事

第一句话就要说清"此事此刻为什么重要"。不要"近年来""随着 AI 的发展""众所周知"。

> **好**："Claude Code 在 2025 年 11 月开始支持 Skills，这套机制把可复用的 agent 行为从插件下沉到了文件系统。"
>
> **坏**："随着大模型应用的不断发展，agent 工程逐渐成为热门话题，本文将探讨……"

### 2. 短句、短段、短块

- 每句 ≤ 20 字（中文按字）。复合句尽量拆。
- 每段 2–4 句，不超过 200 字。
- 单节连续散文 > 200 字时插入小标题、图位或代码块换气。

### 3. 配图只放在认知高点

只在以下情况插图：
- 解释**抽象架构**（多个组件如何协作）。
- 对比**多个方案**（A/B/C 各自长什么样）。
- 展示**时序流程**（请求 → 处理 → 响应）。
- 还原**操作界面**（让读者立刻认出在说哪)。

装饰图、概念封面图、配主题情绪的图：**不要**。

每张图必须有一句话标题（caption），说清图里发生了什么，不能只写"架构图"。

### 4. 引用走文内、不走脚注

正文事实声明都要内联超链接：

```markdown
Anthropic 在 [2025 年 11 月的更新](https://docs.claude.com/en/release-notes/claude-code)
里把 SKILL.md 的位置从插件里抽到了项目根目录。
```

引用翻译文章的原文时，单独一行说清：

```markdown
> 原文：[Building effective agents](https://www.anthropic.com/research/building-effective-agents) by Anthropic, 2024-12
```

### 5. 节奏：3–5 个主章节，每节 ≤ 1200 字

正文层级最多两层：`##` 主章节、`###` 子章节。出现 `####` 说明这一节该拆出来另写一篇。

每个主章节开头一句话回答："**它解决了读者的什么疑问？**" 回答不上来就删掉。

### 6. 代码最小可复现

- 只放足以说明问题的 5–20 行；不放完整工程代码。
- 关键行加注释解释**为什么**这么写，不是这么写**做了什么**。
- 涉及 API key、绝对路径、个人邮箱时用占位符（`$GEMINI_API_KEY` / `<your-path>`）。

### 7. 语气：65% 技术精确 + 35% 可读

技术细节用准确术语（带英文原文），但解释起来用日常说法。不要用"赋能""抓手""链路""闭环"这类商业八股。

参考宝玉的语气：把英文论文里的硬骨头嚼软，但事实层面**不糊**——技术名词、版本号、参数都写准。

### 8. 主观与客观分开标

正文默认是客观陈述。需要表达个人判断或推测时，用引用 callout 显式标出：

```markdown
> 原文说：……（事实，原作者观点）
> 我的看法：……（作者本人观点）
> 分析：……（作者基于事实的推理，非纯主观）
```

混着写会让读者分不清哪是事实哪是判断。

### 9. 文末"延伸阅读"3–5 条

每篇文章必须以 `## 延伸阅读` 收尾。每条加一句话推荐理由：

```markdown
- [Building effective agents](https://www.anthropic.com/research/building-effective-agents) —— agent 设计的第一性原理，写文前先读
- [Simon Willison: Agentic patterns](https://simonw.substack.com/) —— 大量真实工程案例
```

不堆链接、不放没读过的、不超过 5 条。

### 10. 长度分桶

- **短文**：<800 字。讲透一个点子，配 0–1 张图。
- **深度**：1500–2500 字。讲清一个机制，配 2–4 张图。
- **超过 2500 字 → 拆系列**。一次写不完的不要硬塞。

## 反模式（命中立刻改）

| 反模式 | 例子 | 改成 |
|---|---|---|
| 填充过渡 | "在本节中我们将探讨……" | 直接进入论点 |
| 不带出处的判断 | "这种方案显然更好" | "在 [X 测试](url) 中，方案 A 比 B 快 3 倍" |
| 缩写不展开 | "RAG 是主流方案" | "检索增强生成（RAG）是主流方案" |
| 数学先于直觉 | 先抛公式再解释 | 先描述现象，再给公式 |
| 装饰图 | 文章顶图 + 无 caption | 删掉 |
| 自明腔 | "众所周知 LLM 有上下文窗口限制" | "GPT-4 的上下文窗口是 [128K tokens](url)" |
| 长散文块 | 连续 5 段没小标题 | 拆子章节或插图位换气 |
| 商业八股 | "赋能开发者构建闭环" | 写人话 |

## 自检清单（落盘前过一遍）

- [ ] 第一段已经说清"为什么现在读这个有用"。
- [ ] 每个事实声明都有内联出处。
- [ ] 每个判断句都有 callout 标签。
- [ ] 术语首次出现都挂了英文原文。
- [ ] 所有图都有 caption，且在认知高点。
- [ ] 没有连续 >200 字的散文块。
- [ ] 文末有"延伸阅读"，3–5 条。
- [ ] 字数落在目标桶内（短文 <800 / 深度 1500–2500）。
- [ ] 整篇没有"近年来""众所周知""赋能""抓手"等反模式词。
