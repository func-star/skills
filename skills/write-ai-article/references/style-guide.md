# 中文 AI 技术文章风格规范

`write-ai-article` skill 写每一篇文章前过一遍的写作约定。约定不是教条——是这个仓库里文章共同呈现的风格倾向。

## 十条写作约定

### 1. 开头直接抛事

第一句话就说清"此事此刻为什么重要"。回避"近年来""随着 AI 的发展""众所周知"。

> **好**："Claude Code 在 2025 年 11 月开始支持 Skills，这套机制把可复用的 agent 行为从插件下沉到了文件系统。"
>
> **欠妥**："随着大模型应用的不断发展，agent 工程逐渐成为热门话题，本文将探讨……"

### 2. 短句、短段、短块

- 每句尽量 ≤ 20 字（中文按字）。复合句尽量拆。
- 每段 2–4 句，200 字以内。
- 单节连续散文超过 200 字时，插入小标题、图位或代码块换气。

### 3. 配图只放在认知高点

在以下场景插图：
- 解释**抽象架构**（多个组件如何协作）。
- 对比**多个方案**（A/B/C 各自长什么样）。
- 展示**时序流程**（请求 → 处理 → 响应）。
- 还原**操作界面**（让读者立刻认出在说哪里）。

装饰图、概念封面图、配主题情绪的图，回避。

每张图配一句话标题（caption），说清图里发生了什么，不只是"架构图"。

### 4. 引用走文内、不走脚注

正文事实声明配内联超链接：

```markdown
Anthropic 在 [2025 年 11 月的更新](https://docs.claude.com/en/release-notes/claude-code)
里把 SKILL.md 的位置从插件里抽到了项目根目录。
```

翻译类文章的原文，单独一行说清：

```markdown
> 原文：[Building effective agents](https://www.anthropic.com/research/building-effective-agents) by Anthropic, 2024-12
```

### 5. 节奏：3–5 个主章节，每节 ≤ 1200 字

正文层级最多两层：`##` 主章节、`###` 子章节。出现 `####` 通常说明这一节该拆出来另写一篇。

**写作前自检（不写进正文）**：动笔每一节前，心里问一句"这节解决读者什么疑问？"。回答不上来的节考虑删去。

这个自检的结果直接体现在小标题和首段里——好的小标题已经把答案给了，无需在正文里再写一个 `> 解决读者的疑问：……？` 这样的提示块。把内部自检写成显式 callout 会让正文读起来像答疑文档，不像文章。

### 6. 代码最小可复现

- 只放足以说明问题的 5–20 行；不放完整工程代码。
- 关键行加注释解释**为什么**这么写，不是这么写**做了什么**。
- 涉及 API key、绝对路径、个人邮箱时用占位符（`$GEMINI_API_KEY` / `<your-path>`）。

### 7. 现象先于形式化，权衡带反例

- **现象先于形式化**：解释一个概念时，先描述读者能感知的现象，再给定义/公式/伪代码。
- **权衡带反例**：说一个方案好的时候，同时说它在什么场景下不好。只赞不弹的段落补一句边界。
- **语气配比**：约 65% 技术精确 + 35% 通俗解释。技术名词、版本号、参数写准；解释这些名词时用日常说法。回避"赋能""抓手""链路""闭环"这类商业八股。

### 8. 主观与客观分开标

正文默认是客观陈述。需要表达个人判断或推测时，用引用 callout 显式标出：

```markdown
> 原文说：……（事实/原作者观点）
> 我的看法：……（作者本人观点）
> 分析：……（作者基于事实的推理，非纯主观）
```

混着写会让读者分不清哪是事实哪是判断。

### 9. 文末"延伸阅读"3–5 条

每篇文章以 `## 延伸阅读` 收尾。每条配一句话推荐理由：

```markdown
- [Building effective agents](https://www.anthropic.com/research/building-effective-agents) —— agent 设计的第一性原理，写文前先读
- [...](url) —— ……
```

维持 5 条以内，列读过的链接。

### 10. 长度分桶

- **短文**：<800 字。讲透一个点子，配 0–1 张图。
- **深度**：1500–2500 字。讲清一个机制，配 2–4 张图。
- **超过 2500 字**：考虑拆成系列，单篇不硬塞。

## 倾向避开的模式

| 模式 | 例子 | 调整方向 |
|---|---|---|
| 填充过渡 | "在本节中我们将探讨……" | 直接进入论点 |
| 不带出处的判断 | "这种方案显然更好" | "在 [X 测试](url) 中，方案 A 比 B 快 3 倍" |
| 缩写不展开 | "RAG 是主流方案" | "检索增强生成（RAG）是主流方案" |
| 数学先于直觉 | 先抛公式再解释 | 先描述现象，再给公式 |
| 装饰图 | 文章顶图 + 无 caption | 移除 |
| 自明腔 | "众所周知 LLM 有上下文窗口限制" | "GPT-4 的上下文窗口是 [128K tokens](url)" |
| 自问自答式格式块 | 每节开头都来一个 `> 解决读者的疑问：……？` | 把答案直接写进小标题和首段 |
| 长散文块 | 连续 5 段没小标题 | 拆子章节或插图位换气 |
| 商业八股 | "赋能开发者构建闭环" | 用日常说法 |
| 单方面赞歌 | "这个框架完美解决了所有问题" | 补一句它在什么场景下不适用 |
| 训诫腔 | "必须 / 严禁 / 绝对不要……" | 客观陈述事实或后果，不用祈使祈祷的口吻 |

## 自检清单（落盘前过一遍）

- [ ] 第一段已经说清"为什么现在读这个有用"。
- [ ] 每个事实声明都有内联出处。
- [ ] 每个判断句都有 callout 标签。
- [ ] 术语首次出现都挂了英文原文。
- [ ] 所有图都有 caption，且在认知高点。
- [ ] 没有连续 200 字以上的散文块。
- [ ] 文末有"延伸阅读"，3–5 条。
- [ ] 字数落在目标桶内（短文 <800 / 深度 1500–2500）。
- [ ] 没有"近年来""众所周知""赋能""抓手"这类模式词。
- [ ] 提到的每个方案都附带 trade-off 或适用边界。
- [ ] 全文语气不训诫——避免对读者使用"必须 / 严禁 / 绝对"这类训斥腔。
